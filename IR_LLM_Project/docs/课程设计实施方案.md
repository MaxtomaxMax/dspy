# 基于 DSPy 的代码库多跳检索课程设计实施方案

## 1. 项目目标

本课题围绕 `DSPy` 源码库，构建两套推理框架一致、检索后端不同的多跳代码问答系统，并基于统一 benchmark 做实证对比：

- 路线 A：传统向量检索基线 `Dense RAG`
- 路线 B：基于 `ripgrep` 的精确匹配智能体 `Grep Agent`

当前项目的第一阶段已经基本完成：

- `DSPy_Multihop_QA.json` 已构建完成
- `IR_LLM_Project/src/rag_baseline/` 中已经具备一套可运行的 dense baseline

## 2. 对现有阶段一实现的兼容性判断

阶段一与最初设想有一些实现细节偏差，但不需要返工。

第一，当前 dense baseline 不是 `LangChain + Chroma` 组合，而是：

- 结构化代码切块
- 本地 embedding 编码
- numpy brute-force 检索
- `vector_search + open_chunk` 双工具 ReAct

这仍然属于传统向量检索范式，而且对代码问答更合理，可以直接作为正式基线。

第二，当前 dense baseline 的工具接口天然是两阶段的：

- `vector_search(query)`
- `open_chunk(chunk_id)`

因此 grep 路线也应保持同样的“先定位，再深读”思想，而不是只暴露单一搜索接口。这样两套系统在实验上更可比。

本项目后续实现采用以下兼容性原则：

- 不改动 `rag_baseline` 现有实现
- grep 方案全部放在新的 `IR_LLM_Project/src/grep_agent/` 目录中
- 两套系统共用同一 benchmark、同一 LM 配置方式、同一结果记录结构

## 3. 总体技术路线

### 3.1 阶段一：传统向量检索基线

当前 dense baseline 的角色是正式对照组，核心特征如下：

- 语料范围限定为 `dspy/**/*.py`
- 采用结构化代码切块
- 使用代码 embedding 做本地向量化
- 通过 DSPy `ReAct` 封装成可多跳调用的 agent
- 批量运行结果记录为 `AnswerRunRecord`

### 3.2 阶段二：精确匹配智能体

阶段二是本课题的核心创新点。系统不依赖向量数据库，而是让大模型通过底层代码检索工具直接导航源码。

建议的原子工具为：

1. `global_search(regex_pattern: str) -> str`
   使用 `rg` 在 `dspy/**/*.py` 中做全局正则匹配，只返回文件路径、行号和少量匹配行摘要。
2. `read_file_lines(filepath: str, start_line: int, end_line: int) -> str`
   在模型锁定候选文件后，再读取具体代码段。

### 3.3 阶段三：DSPy ReAct 管道组装

两套系统应尽量保持共形：

- `DenseRAGAgent`
- `GrepCodeAgent`

二者都使用：

- 相同的 `dspy.ReAct`
- 相同的 `max_iters`
- 相同的 `question -> answer` 输出接口
- 相同的运行结果记录字段

### 3.4 阶段四：自动化量化评测

最终评测建议至少覆盖：

- 最终答案质量
- 工具调用次数
- token 消耗量
- 检索链覆盖情况
- 单题耗时

建议统一记录的字段包括：

- `pred_answer`
- `trajectory`
- `tool_calls`
- `token_usage`
- `used_files`
- `elapsed_seconds`

现有 `AnswerRunRecord` 已具备这套结构，因此 grep 路线可以直接复用。

## 4. 对状态维护位置的建议

当前更推荐采用“Tool-level state 为主，Agent-level state 为辅”的设计。

原因是阶段二最关键的两个安全风险分别是：

- 输出过长导致 token 溢出
- ReAct 陷入重复搜索死循环

这两类问题都由工具 I/O 直接触发，因此应优先放在工具内部处理：

- 工具返回值一产生就立刻截断
- 工具执行前先判断是否为连续重复请求
- 即使后续更换 agent、prompt 或 LM，这层保护仍然存在

Agent 外层状态依然有价值，但更适合负责：

- 每题最大工具调用次数
- 总上下文预算
- 失败早停和日志提取

因此本项目建议采用的分层是：

- 工具层：重复调用拦截、字符截断、路径越界检查、行数钳制
- Agent 层：回合预算、全局早停、轨迹统计

## 5. 新增 grep 方案的工程结构

为避免影响现有 dense baseline，实现建议全部放在新的目录：

- `IR_LLM_Project/src/grep_agent/config.py`
- `IR_LLM_Project/src/grep_agent/grep_tools.py`
- `IR_LLM_Project/src/grep_agent/agent.py`
- `IR_LLM_Project/src/grep_agent/run_benchmark.py`
- `IR_LLM_Project/src/grep_agent/search_grep.py`

各模块职责如下：

- `config.py`
  管理 grep 路线默认路径与安全阈值
- `grep_tools.py`
  封装 `global_search` 和 `read_file_lines`
- `agent.py`
  定义 `GrepCodeAgent`
- `run_benchmark.py`
  批量运行 grep agent
- `search_grep.py`
  单独调试 grep 工具

## 6. 阶段二的工程要求

### 6.1 `global_search`

实现要求：

- 底层调用 `rg`
- 搜索范围限制在 `dspy/` 下的 Python 文件
- 返回 JSON 字符串，字段至少包含：
  - `pattern`
  - `results`
  - `file_path`
  - `line_number`
  - `message`

安全要求：

- 空模式直接报错
- 非法正则直接报错
- 返回字符数超过阈值时截断
- 若连续两次发出完全相同的搜索请求，则直接拦截

### 6.2 `read_file_lines`

实现要求：

- 只允许读取 `dspy/` 下的 `.py` 文件
- 输出带行号的代码片段
- 返回 JSON 字符串

安全要求：

- 非法路径直接拒绝
- 超出文件范围时返回清晰提示
- 单次最多读取固定行数
- 若连续两次读取完全相同的文件区间，则直接拦截

## 7. 阶段三四的后续推进建议

阶段二完成后，建议按这个顺序继续推进：

1. 用 `grep_agent.run_benchmark` 跑通 grep agent 的批量问答
2. 复用现有 `rag_baseline.eval_answers` 先做本地 `EM/F1` 粗评测
3. 补一个 `LLM-as-a-Judge` 脚本作为正式准确率指标
4. 汇总 dense 与 grep 两套系统的：
   - 准确率
   - 平均工具调用次数
   - 平均 token 开销
   - 平均耗时

## 8. 当前结论

现阶段最合理的策略不是返工阶段一，而是：

- 将阶段一保留为正式 dense baseline
- 在新的 `grep_agent` 目录中实现阶段二 grep agent
- 将状态维护优先放在工具内部
- 在日志格式上主动向阶段三四对齐

这条路线改动最小，兼容性最好，也最适合后续课程论文写作和复现实验。
