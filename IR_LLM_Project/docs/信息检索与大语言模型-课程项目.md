# 开题报告

课程题目：**基于大语言模型的代码库多跳检索增强系统研究：精确匹配与向量检索的实证对比**

## 课题意义

检索增强生成（RAG, Retrieval-Augmented Generation）技术已成为解决大语言模型（LLM）领域知识匮乏与幻觉问题的核心范式。然而，当前学术界与工业界的主流框架过度依赖基于稠密向量（Dense Vector）的语义检索机制。在面对诸如底层代码库、高结构化日志等强调符号逻辑与严格命名的特定垂直领域时，向量检索常因词汇的语义近似性而导致召回精度下降，进而引发上下文窗口污染与推理迷失。

近年来，以 Claude Code、Cursor 为代表的前沿工业级代码智能体逐渐屏弃厚重的向量数据库，转而采用以 Grep、抽象语法树（AST）解析为代表的轻量级、确定性字符匹配技术。本项目旨在通过构建对比实验，系统性地探究在面向代码库的多跳检索问答场景下，精准字符匹配检索与传统向量语义检索对大模型多跳推理（Multi-Hop Reasoning）能力的具体影响。本课题不仅有助于反思当前 RAG 系统设计的盲目“向量化”倾向，证明轻量级系统工具在特定高逻辑密度场景下的优越性，还能为构建低成本、高可靠性的垂直领域自律智能体提供坚实的实证数据与技术指导。

## 研究目标与工作内容

本项目的核心目标是依托 DSPy 声明式大语言模型编程框架，设计并实现两套架构平行但检索底层逻辑截然不同的多跳代码问答智能体，并通过量化指标评估两者在复杂任务环境下的性能差异。

具体工作内容包含四个核心环节。（1）基准测试集（Benchmark）的自动化构建：项目将选取 DSPy 框架的官方 GitHub 开源代码库作为目标知识源，生成包含跨文件继承调用、精准符号追踪等复杂逻辑的三十至五十道多跳问答测试用例，并输出包含标准答案与检索路径的结构化数据。（2）基线系统的搭建，项目将基于文本切分、嵌入模型（Embedding）与本地向量数据库构建传统的向量检索工具，并将其接入 DSPy 的 ReAct 框架中，形成向量驱动的多跳智能体。（3）创新检索系统的实现：项目将通过 Python 封装操作系统底层的精确匹配工具（如 ripgrep），设计包含结果截断与防死循环逻辑的安全检索工具，同样将其赋予 DSPy 智能体以实现基于精准匹配的多跳推理。（4）评估与分析工作：项目将开发自动化评测脚本，综合对比系统在最终回答准确率、平均检索跳数以及 Token 消耗量上的具体表现。

## 相关工作与参考文献

（1）检索增强生成（RAG）与多跳推理机制

大语言模型在处理垂直领域和长尾知识时常面临幻觉问题。自 Lewis 等人[@lewisRetrievalAugmentedGenerationKnowledgeIntensive2020]系统性提出检索增强生成（RAG）范式以来，引入外部知识库已成为事实标准。然而，传统 RAG 多采用“单次检索-生成”模式，在面对复杂逻辑任务时表现不佳。为了解决这一问题，学术界提出了多跳检索与推理架构。例如，Khattab 等人提出的 Baleen 系统[@khattabBaleenRobustMultiHop2021]论证了在复杂问答中多跳管道的必要性；Yao 等人提出的 ReAct 框架[@yaoReActSynergizingReasoning2022]则将推理（Reasoning）与行动（Acting）结合，赋予了模型主动调用工具进行多轮环境交互的能力。在本课题的代码问答场景中，理解跨文件的方法调用、类继承等结构，本质上是一个典型的高度依赖上下文链路的多跳检索问题。

（2）从提示词工程到声明式大语言模型编程

随着 RAG 和智能体管道日益复杂，传统基于硬编码提示词模板的开发模式（如 LangChain、LlamaIndex 初期的主流做法）暴露出极度脆弱、难以跨模型泛化等问题。为此，Khattab 等人提出了 DSPy 框架[@khattabDSPyCompilingDeclarative2023]，将 LLM 管道抽象为计算图，提出“声明式编程”与“自动编译优化”的概念。DSPy 摒弃了手工编写长提示词，转而通过 Teleprompter 在小样本上自动引导（Bootstrap）和优化提示词或微调权重，其思想受到 PyTorch 等深度学习计算图框架的启发。在 DSPy 发布之后，学术界在“程序化优化提示词”方向上进行了大量延伸探索，如 TextGrad [@yuksekgonulOptimizingGenerativeAI2025]引入了基于 LLM 反向传播的文本梯度优化机制。本课题引入 DSPy 框架作为基座，不仅是为了构建 ReAct 智能体，更重要的是利用其自动编译能力，屏蔽手工调优提示词对实验对比造成的干扰，从而公平、纯粹地对比不同检索底层的真实效能。

（3）代码智能中的检索策略：语义向量化与确定性匹配

在垂直领域的 RAG 实践中，过度依赖密集向量检索的弊端正逐渐显现。在自然语言场景下高效的 Embedding 模型，在处理包含高度符号逻辑、自定义变量名与严密语法树的代码库时，常因“词汇表外问题（OOV）”和“语义平滑”而导致精确匹配失败。为此，业界探索了针对代码的特定检索方案。例如，基于 AST 解析的代码理解工作（如 GraphCodeBERT[@guoGraphCodeBERTPretrainingCode2020]）证明了语法结构信息的重要性。

更为显著的是，2024 年以来以 SWE-agent[@yangSWEagentAgentComputerInterfaces2024]、Devin 为代表的前沿工业级代码智能体，在解决真实 GitHub 仓库级 issue 时，其底层检索逻辑大规模转向了传统的操作系统工具（如 `grep`, `find`, `cat`）以及语言服务器协议（LSP）。SWE-agent 的实证研究表明，赋予 LLM 精确的字符串匹配与文件导航接口，远比单纯的向量检索更能提高代码修复的成功率。Claude Code 等最新工具也通过系统级的 Grep 搜索代替了向量数据库的预索引环节，不仅消除了向量化的高昂计算与存储成本，也规避了上下文窗口被不相关近似代码污染的风险。本课题正是对这一行业前沿范式转变的系统性实证响应。

## 基本技术路线

本项目的技术路线呈现典型的对比消融实验特征，整体分为数据准备、双轨系统开发与量化评测三个阶段。在第一阶段，首先克隆 DSPy 源码库并锁定特定 Commit 版本以保证实验的可复现性，随后基于代码库生成符合要求的 JSON 格式多跳问答评测集。在第二阶段，进行双轨工具链的开发：针对向量基线系统，运用 LangChain 的递归字符切分器对源码进行处理，利用轻量级嵌入模型将数据向量化并存入本地向量数据库，封装为语义检索工具；针对精准匹配系统，利用 Python 的 subprocess 模块调用操作系统的 ripgrep 命令，通过正则表达式参数控制匹配精度，并增加基于行数的强截断机制以防止 Token 溢出，封装为字符匹配工具。随后，利用 DSPy 内置的 `dspy. ReAct` 模块，分别将两种工具注入，构建出具备“思考-检索-再思考”能力的多跳查询管道。在第三阶段，编写批处理测试框架，依次向两个系统输入生成的基准测试问题，通过 DSPy 的度量机制记录每一次交互的日志追踪（Trace），运用高精度语言模型比对生成答案与标准答案的语义一致性，最后通过脚本清洗日志，汇总输出准确率与效率对比的图表数据，完成技术闭环。
